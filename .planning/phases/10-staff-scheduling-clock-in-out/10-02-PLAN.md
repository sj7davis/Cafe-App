<!-- PIN authentication pattern for the clock page -->
The clock page does NOT use the staff JWT stored in localStorage.
Staff are anonymous tablet users who enter a PIN.
The backend must look up staffId by PIN. This requires a new public tRPC procedure on clockRouter:
  clockRouter.clockInByPin: input { slug: string, pin: string, note?: string }
  → resolves venueId from slug (query venues table)
  → finds staffAccount where venueId matches and clockPin matches
  → runs the same clockIn logic (double-clock-in guard, insert event)
  → returns { ok, staffName, clockedAt, penaltyFlag, staffToken } where staffToken is the staff JWT
    (so the page can subsequently call getMyStatus with the real token for subsequent break/out actions)

SIMPLER APPROACH (use this): Store the staffToken in component state after PIN entry.
  Step 1: clockRouter.loginByPin: input { slug: string, pin: string }

    → finds staffAccount where venueId=venueId AND clockPin=pin AND isActive=true
    → generates a staff JWT (same fields: staffId, venueId, role, name)
    → returns { token, staffName, staffId }
  Step 2: Once the component has the token, it can call clock.getMyStatus, clock.clockIn, clock.clockOut, etc.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public tablet → loginByPin | No session; PIN is the only credential. Token issued is short-lived (8h). |
| loginByPin token → clock mutations | Subsequent clock actions use the 8h JWT; the tablet never stores owner credentials |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04 | Spoofing | loginByPin PIN brute force | accept | 4-digit PIN on a shared tablet is comparable to a door code; no rate-limiting required for v2.2 cafe context |
| T-10-05 | Information Disclosure | clockPin stored plain text | accept | PIN is low-value; bcrypt overhead not warranted for a 4-digit access code |
| T-10-06 | Elevation of Privilege | 8h JWT from loginByPin | mitigate | JWT only grants staffId/venueId — not owner role. Clock mutations verify venueId from JWT matches event venue. |
| T-10-07 | Denial of Service | Inactivity reset | mitigate | 30-second inactivity timer resets screen; next staff member is not locked out by a lingering session |
| T-10-SC | Tampering | npm installs | accept | No new packages in this plan |
</threat_model>
